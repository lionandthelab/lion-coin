# 니모닉·개인키 보안 원칙

이 문서는 트랙 A(테스트넷 지갑 개발) 진행 중 니모닉·개인키를 다룰 때 지켜야 할 원칙을 정리한다. `비트코인_획득_전략_및_개발제안서.md` 2-4절(완료 기준)과 6절(리스크·보안)의 내용을 실행 가능한 규칙으로 구체화한 것이다.

## 1. 핵심 원칙

1. **니모닉은 종이에만 기록한다.** 코드 파일, 커밋, 로그, 클라우드 노트, 스크린샷, 메신저 어디에도 남기지 않는다.
2. **학습용 코드(`src/`)로 실자산을 관리하지 않는다.** 이 저장소의 지갑 코드는 테스트넷/signet 전용이며, 실제 가치를 지닌 니모닉을 여기서 생성·파생하지 않는다.
3. **실제 sats 수령은 검증된 라이트닝 지갑(트랙 B, 예: Blink)으로만 한다.** 이 저장소가 개인키를 직접 보유하지 않는다.
4. **커스터디얼 지갑에는 소액만 둔다.** "not your keys, not your coins" — 목표 금액(약 600 sats) 이상을 장기 보관하지 않는다.

## 2. 이 저장소의 기술적 강제 장치

- `.gitignore`가 `.env`, `*.mnemonic`, `secrets/`를 제외한다 — 실수로 커밋되는 것을 1차 방어.
- `.claude/settings.json`의 `permissions.deny`가 `Read(./.env)`, `Read(./secrets/**)`를 막는다 — 에이전트가 실수로 비밀 파일 내용을 로그·응답에 노출하는 것을 방지.
- `src/wallet.js`의 `NETWORKS` 기본값은 `testnet` — 네트워크를 명시적으로 `mainnet`으로 바꾸지 않는 한 항상 테스트넷 주소(`tb1q...`)가 파생된다.
- 니모닉을 다루는 모든 예시 명령(`harness/ACTION_REQUIRED.md`의 A3 항목 등)은 터미널 화면 출력만 사용하고, 파일로 저장하는 단계를 포함하지 않는다.

## 3. 니모닉이 필요한 순간의 절차 (예: A3 faucet 수령)

1. 로컬 터미널에서 `deriveAddress`/`generateMnemonic`을 호출해 화면에만 출력한다.
2. 니모닉 24단어를 종이에 옮겨 적는다. 사진·타이핑·클립보드 복사 금지.
3. 터미널 스크롤백을 지우거나 터미널을 닫는다.
4. `harness/state.json`의 관련 task `notes`에는 **주소만** 기록한다. 니모닉 문자열은 절대 기록하지 않는다.
5. 종이는 사람이 직접 보관한다. 에이전트는 이 단계에 관여하지 않는다.

## 4. 노출 사고 발생 시

니모닉이 코드·저장소·로그·클라우드 등 통제되지 않은 채널에 노출된 것을 발견하면:

1. 해당 니모닉으로 파생된 주소에 실자산이 없는지 확인한다 (테스트넷 전용이므로 통상 실자산 없음).
2. 노출된 텍스트가 git 히스토리에 있다면 즉시 사용자에게 보고한다 — 히스토리 재작성(rebase/force-push)은 파괴적 작업이므로 에이전트가 임의로 수행하지 않는다.
3. 실자산과 연결된 적이 있는 니모닉이라면 즉시 사용자에게 알리고, 해당 지갑에서 자산을 새 니모닉으로 이전하도록 권고한다.
4. 재발 방지를 위해 `.gitignore`/`permissions.deny` 패턴에 누락이 있었는지 점검한다.

## 5. 체크리스트 (커밋 전)

- [ ] 커밋에 24단어(또는 12단어) 니모닉 패턴이 포함되지 않았는가
- [ ] 커밋에 WIF/hex 형식의 개인키가 포함되지 않았는가
- [ ] `.env`, `secrets/`, `*.mnemonic`이 스테이징에 없는가 (`git status`로 확인)
- [ ] 로그(`logs/*.md`)에 실제 잔액 조회용 API 키·엔드포인트 응답 원문이 그대로 붙지 않았는가
