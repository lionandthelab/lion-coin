# 비트코인 첫 1,000원 획득 프로젝트 — 전략 및 개발 제안서

**프로젝트명:** Satoshi Zero-to-One (첫 사토시 획득 프로젝트)
**목표:** 외부 구매 없이, 직접 만든 지갑으로 직접 만든 제품을 팔아 약 1,000원어치(수백 sats)의 비트코인을 실제로 수령한다.
**작성일:** 2026-07-14

---

## 1. 핵심 전략 요약

### 1-1. 가장 중요한 기술적 판단: 온체인이 아니라 라이트닝(Lightning)

1,000원어치 비트코인은 **온체인(L1) 거래로는 사실상 불가능**합니다. 온체인 트랜잭션 수수료가 수령 금액보다 커지는 "먼지(dust)" 구간이기 때문입니다. 따라서 이 프로젝트의 결제·수령 레이어는 처음부터 **라이트닝 네트워크(L2)** 로 설계해야 합니다.

- 라이트닝은 수수료가 1 sat 미만~수 sat 수준이라 소액 수령에 적합
- 결제가 즉시 확정되어 디지털 상품 판매와 궁합이 좋음
- 온체인 지갑 개발은 "학습 트랙"으로, 라이트닝은 "수익 트랙"으로 분리

### 1-2. 3-트랙 병행 전략

| 트랙 | 목적 | 산출물 |
|---|---|---|
| **A. 개발 학습 트랙** | 지갑·트랜잭션을 코드로 완전히 이해 | 테스트넷 HD 지갑 + 서명·전송 CLI |
| **B. 수익 트랙 (본선)** | 실제 sats 수령 | 라이트닝 결제 페이지 + 디지털 상품 1종 |
| **C. 벤치마킹 트랙** | 팔리는 상품 구조 분석 | 벤치마킹 리포트 + 상품 기획서 |

핵심 논리: **A는 배우기 위해, B는 벌기 위해.** 온체인 지갑을 직접 짜는 것은 훌륭한 학습이지만, 실제 1,000원 수령은 라이트닝 인보이스 하나로 끝나야 합니다.

## 2. 트랙 A — 지갑·거래 개발 (테스트넷)

### 2-1. 목표

"지갑 생성 → 주소 파생 → 코인 수령 → 트랜잭션 서명·전송"의 전 과정을 코드로 직접 수행하고 원리를 이해한다. **반드시 테스트넷(testnet4 또는 signet)에서 진행**하며, 실제 자산 키를 이 코드로 관리하지 않는다.

### 2-2. 기술 스택

- **언어/라이브러리:** Node.js + `bitcoinjs-lib` + `bip39` + `bip32` (또는 Python `bitcoinlib`)
- **블록체인 조회/브로드캐스트:** mempool.space API 또는 Blockstream Esplora API (자체 노드 불필요)
- **테스트 코인:** signet/testnet faucet에서 무료 수령

### 2-3. 구현 범위 (약 3~5일)

```
satoshi-lab/
├── src/
│   ├── wallet.js      # BIP39 니모닉 생성 → BIP84 (bc1 네이티브 세그윗) 파생
│   ├── balance.js     # Esplora API로 UTXO 조회
│   ├── send.js        # PSBT 구성 → 서명 → 브로드캐스트
│   └── fees.js        # 수수료 추정 (sat/vB)
└── cli.js             # create | address | balance | send 명령
```

핵심 코드 스케치 (지갑 생성):

```javascript
const bip39 = require('bip39');
const { BIP32Factory } = require('bip32');
const ecc = require('tiny-secp256k1');
const bitcoin = require('bitcoinjs-lib');

const mnemonic = bip39.generateMnemonic(256);          // 24단어
const seed = bip39.mnemonicToSeedSync(mnemonic);
const root = BIP32Factory(ecc).fromSeed(seed, bitcoin.networks.testnet);
const child = root.derivePath("m/84'/1'/0'/0/0");      // BIP84 testnet
const { address } = bitcoin.payments.p2wpkh({
  pubkey: child.publicKey, network: bitcoin.networks.testnet
});
console.log({ mnemonic, address });                    // tb1q...
```

### 2-4. 완료 기준 (Definition of Done)

1. 니모닉으로부터 결정적으로 동일한 주소가 파생됨을 검증
2. Faucet 코인을 수령하고 UTXO를 API로 확인
3. 다른 자기 주소로 PSBT 서명·전송 성공, 익스플로러에서 확인
4. 니모닉·개인키의 보안 원칙(오프라인 백업, 코드 저장 금지) 문서화

## 3. 트랙 B — 라이트닝 수익 파이프라인 (실전)

### 3-1. 수령 인프라 선택지

| 옵션 | 방식 | 난이도 | 추천도 |
|---|---|---|---|
| **커스터디얼 라이트닝 지갑** (Blink 등) | 앱 설치 후 즉시 인보이스 발급 | ★ | 1단계용 최적 |
| **LNbits (호스팅)** | 계정형 라이트닝 지갑 + API/결제링크 | ★★ | 개발자에게 적합 |
| **BTCPay Server (자체 호스팅)** | VPS에 결제 프로세서 직접 운영 | ★★★★ | 확장 단계용 |

**권장:** 1단계는 커스터디얼 지갑 또는 LNbits로 "첫 sats"를 빠르게 받고, 이후 BTCPay Server를 자체 구축해 셀프커스터디로 이전. (커스터디얼은 소액 한정으로만 사용 — "not your keys, not your coins.")

### 3-2. 결제 페이지 개발 (약 2~3일)

- LNbits의 **LNURL-pay / Pay Link** 확장 또는 BTCPay의 결제 버튼을 사용
- 정적 페이지(자체 도메인 또는 GitHub Pages) + 결제 완료 webhook → 디지털 파일 자동 전달
- 흐름: 상품 페이지 → "500 sats 결제" 버튼 → 인보이스 QR → 결제 확인 → 다운로드 링크 발급

### 3-3. 첫 수익까지의 현실적 경로 (빠른 순)

1. **Stacker News 등 sats 보상형 커뮤니티에 글 게시** — 개발 과정 자체를 영어로 연재하면 tips 수령 가능. 콘텐츠 제작 경험을 그대로 활용
2. **자기 결제 페이지에 지인/커뮤니티 첫 구매 유도** — 500~1,000 sats 가격의 초저가 디지털 상품
3. 마이크로태스크·게임형 앱은 시간 대비 효율이 낮아 **비추천** (다만 인보이스 테스트용으로는 유용)

## 4. 트랙 C — 상품 벤치마킹 및 기획

### 4-1. 벤치마킹 대상 (비트코인으로 실제 팔리는 것들)

비트코인 결제 시장에서 실제로 거래되는 상품 유형은 다음과 같이 정리됩니다:

- **디지털 콘텐츠 유료화:** 페이월 방식으로 글·영상·파일에 sats 과금 (Paywall 계열, Stacker News의 유료 글)
- **디지털 아트/사진 판매:** 사진·일러스트를 sats로 판매하는 마켓
- **개발자 도구·템플릿:** 스크립트, 노션 템플릿, 프롬프트 팩, 전자책
- **프리랜서 서비스:** BTC로 급여·대금을 받는 원격 작업 (Bitwage 등 경유)
- **기프트카드 리셀 생태계:** Bitrefill 등 — 판매보다는 sats 사용처로 참고

### 4-2. 판매 상품 후보 (보유 자산 재활용 — 제작비 0원)

| 후보 | 형태 | 가격안 | 강점 |
|---|---|---|---|
| **"Bitcoin Wallet from Scratch" 미니 가이드** | PDF/영문 | 1,000 sats | 트랙 A 산출물을 그대로 상품화. Stacker News 독자층과 정합 |
| **개발자용 프롬프트/템플릿 팩** | ZIP | 500 sats | 기존 개발·콘텐츠 노하우 재활용 |
| **교육용 인쇄물 템플릿 (영문화)** | PDF | 500 sats | 기존 디자인 자산 재활용 |

**권장 1호 상품:** 미니 가이드. "만드는 과정 = 마케팅 콘텐츠 = 상품"이 되는 삼중 구조라 별도 마케팅 비용이 없습니다.

### 4-3. 벤치마킹 리포트 작성 항목

경쟁 상품 5개에 대해: 가격(sats), 결제 방식(온체인/LN), 전달 방식, 페이지 구성, 후기·판매 신호를 표로 정리 → 가격·구성 결정에 반영.

| **1주차 말** | 라이트닝 지갑 개설, 첫 인보이스 발급·자가 테스트 | 수령 가능한 LN 주소 |

| **3주차** | **첫 1,000원(약 500~700 sats) 수령 확인** | 결제 내역 스크린샷 = 프로젝트 완료 |

## 6. 리스크 및 유의사항

**규제·세무 (한국)**
- 개인이 재화·용역 대가로 가상자산을 받는 것 자체는 가능하나, 반복적 판매 소득은 사업소득으로 과세 대상이 될 수 있음 — 기존 프리랜서 소득과 함께 종합소득 신고 시 반영 필요
- 국내 거래소로 옮겨 원화화할 경우 실명계좌·KYC 필수, 거래소별 개인지갑 출금 시 지갑 사전등록(화이트리스팅) 정책 확인 필요
- 가상자산 양도차익 과세 시행 일정은 변동이 있어 왔으므로, 원화 환전 시점에 최신 규정 확인 권장

**보안**
- 니모닉은 종이에만 기록, 코드·클라우드·스크린샷 금지
- 학습용 코드(트랙 A)로 실제 자산 관리 금지 — 실자산은 검증된 지갑 사용
- 커스터디얼 지갑에는 소액만 보관

**사업적 리스크**
- 소액 sats 판매는 수익성보다 **학습·포트폴리오 가치**가 본질 — "비트코인 결제 시스템을 직접 구축·운영해본 개발자"라는 실적은 향후 외주(결제 연동, 핀테크)에서 차별화 요소가 됨
- 파우셋·리워드 앱류는 시간당 기대수익이 극히 낮으므로 주력으로 삼지 않음

## 7. 예상 비용 및 결론

- **금전 비용:** 0원 (호스팅 무료 티어 + faucet + 커스터디얼 지갑 기준. BTCPay 자체 호스팅 시 VPS 월 5~10달러)
- **시간 비용:** 약 3주, 주당 8~10시간
- **기대 성과:** 첫 sats 수령(정량 목표) + 온체인/라이트닝 풀스택 이해 + 영문 기술 콘텐츠 포트폴리오 + 재판매 가능한 디지털 상품 1종

> **결론:** "1,000원어치 비트코인"이라는 목표는 라이트닝 위에서만 경제적으로 성립합니다. 온체인 지갑 개발은 테스트넷 학습 트랙으로 확실히 마스터하고, 실제 수익은 **라이트닝 인보이스 + 초저가 디지털 상품 + 과정의 콘텐츠화**라는 삼각 편대로 달성하는 것을 제안합니다.
